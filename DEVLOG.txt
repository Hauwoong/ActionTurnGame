- 카드 효과 처리 로직 설계 및 테스트

### 문제
- CardData 추상 클래스와 virtual / abstract 혼용으로 컴파일 오류 발생
- BattleContext 전달 구조 설계 미흡으로 UseCard 호출 오류 발생

### 해결
- CardData 구조를 추상 기반 공통 처리 + 데이터 중심 구조로 재설계
- 카드 사용 시 BattleContext를 통해 전투 정보 전달하도록 구조 수정
- 카드 타입 분류(enum) 제거 → 주사위 데이터 중심 구조로 통합
- 기본 카드와 특수 카드 상속 구조 방향 정립
- 카드 역할을 주사위 조합으로 표현하는 방식 확정

### 배운 점
- 카드 시스템의 핵심은 “카드 타입”이 아니라 “행동 데이터”이다
- ScriptableObject + 데이터 중심 구조는 확장성이 매우 높다
- abstract / virtual은 역할이 명확할 때만 사용하는 것이 좋다
- BattleContext 같은 컨텍스트 객체는 시스템 결합도를 낮춰준다
- 중복 구조를 의심하고 제거하는 설계가 장기 유지보수에 중요하다

-합과 일방공격 로직이 분리되어 코드 중복 및 유지보수 부담 증가
-Dice 저장 구조가 단순 Queue 자료구조에 머물러 설계 의도 불명확
-ResolutionSystem에 책임이 과도하게 집중되는 God Script화 위험
-RNG 내부 생성 방식으로 인해 리플레이 / 동기화 확장성 부족

###해결

-CharacterRuntime 재생성 방식 제거 → 동일 객체 유지 구조로 안정화
-DiceQueue 기반 단일 Resolve 루프 설계로 합 / 일방공격 통합
-“상대 Dice 존재 여부”에 따른 분기 처리 구조 확정
-CombatLog를 Turn 단위 단일 리스트로 통합 관리
-CombatEvent 기반 이벤트 기록 구조 채택
-RNG 외부 주입 방식 설계 방향 확정
-ResolutionSystem에서 ClashEngine 분리 설계
-DiceEngine은 계산 전용, CombatEngine은 상태 변경 담당으로 역할 명확화

###배운 점

-전투 시스템의 핵심은 “Dice 이동”이 아니라 “Dice 소비 흐름”이다
-합과 일방공격은 다른 시스템이 아니라 동일 루프의 분기다
-전투 중심 객체는 카드가 아니라 CharacterRuntime이다
-슬롯은 전투 객체가 아니라 순서 결정자이다
-로그 기반 설계는 디버깅과 확장성 모두에 유리하다
-엔진이 커질수록 계산 시스템과 상태 변경 시스템을 분리해야 한다

###구조적 결론

-전투 중심 = CharacterRuntime
-슬롯 = 전투 전 매칭 / 속도 결정 전용
-로그 = Turn 단위 단일 리스트
-Dice = 이동 개념이 아니라 소비 기록 개념

###문제

-Resolve 중 Runtime 생성으로 참조 불안정 발생 가능성

-1:1 매칭 기반 합 구조의 확장성 한계

-일방 공격 처리 로직 분리

-전투 계산 시스템과 연출 시스템 완전 분리

###배운 점

-전투는 소비가 아니라 “진행” 구조로 보는 것이 맞다
