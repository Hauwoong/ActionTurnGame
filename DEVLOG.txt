-HP / Energy UI 갱신 타이밍 구조에 대한 혼란
-UI 통합 관리 vs 개별 관리 구조 설계 고민 발생

-UI 배치는 좌표보다 Anchor 구조가 훨씬 중요하다
-UI를 기능별로 분리하면 확장성과 가독성이 크게 좋아진다

## 2026-01-28

### 오늘 한 것
- ScriptableObject 기반 카드 데이터(CardData) 구조 설계
- 카드 비용 / 데미지 / 설명 / 이미지 데이터 분리
- Card UI 프리팹 제작 및 자동 생성 시스템 구현
- Horizontal Layout Group 기반 카드 정렬 구조 완성
- CardUIManager를 통한 카드 관리 구조 설계
- 카드 클릭 → BattleManager → 전투 처리 흐름 연동
- TextMeshPro 한글 폰트 적용 및 UI 텍스트 정상 출력 처리

### 문제
- CardUI 프리팹 Missing 오류로 카드가 생성되지 않음
- 카드 클릭 시 Player / Enemy 참조 구조 혼란 발생
- 이벤트(Player.OnHandChanged) 직접 호출 시 컴파일 오류 발생
- TextMeshPro 기본 폰트에서 한글 출력 불가 문제 발생
- UI Layout 적용 후 카드 위치가 예상과 다르게 배치됨

### 해결
- CardUI 프리팹 재연결 및 Inspector 구조 정리
- 카드 사용 로직을 BattleManager 중심 구조로 재설계
- 이벤트 구조 수정(Invoke → 내부 호출 방식으로 변경)
- 한글 지원 폰트(SDF) 적용으로 텍스트 깨짐 해결
- Layout Group + RectTransform 설정 재조정

### 배운 점
- ScriptableObject는 데이터와 로직 분리에 매우 효과적이다
- 카드 시스템은 중앙 관리자(BattleManager) 중심 구조가 안정적이다
- 이벤트는 외부에서 직접 호출하면 구조가 깨질 수 있다
- UI 자동 배치는 Layout 시스템 이해가 필수이다
- 폰트 세팅도 UI 품질에 큰 영향을 준다

## 2026-01-29

### 오늘 한 것
- ScriptableObject 기반 CardData 추상 클래스 구조 설계
- 주사위(DiceData) 기반 카드 효과 처리 구조 설계
- 카드 → 주사위 → 전투 처리 흐름 설계
- Library of Ruina 스타일 카드 구조 방향성 확정
- enum 기반 카드 타입 제거 → 상속/데이터 중심 구조로 전환
- BattleContext 기반 전투 정보 전달 구조 설계
- CardUI ↔ BattleManager ↔ CardData 연동 구조 개선
- 카드 효과 처리 로직 설계 및 테스트

### 문제
- CardData 추상 클래스와 virtual / abstract 혼용으로 컴파일 오류 발생
- BattleContext 전달 구조 설계 미흡으로 UseCard 호출 오류 발생
- CardData.Use / Resolve 구조 혼란으로 인한 타입 불일치 오류
- 다이스 타입(DiceType) 처리 방식 설계 혼란
- 카드 효과 책임 분리 기준에 대한 구조적 고민 발생

### 해결
- CardData 구조를 추상 기반 공통 처리 + 데이터 중심 구조로 재설계
- 카드 사용 시 BattleContext를 통해 전투 정보 전달하도록 구조 수정
- 카드 타입 분류(enum) 제거 → 주사위 데이터 중심 구조로 통합
- 기본 카드와 특수 카드 상속 구조 방향 정립
- 카드 역할을 주사위 조합으로 표현하는 방식 확정

### 배운 점
- 카드 시스템의 핵심은 “카드 타입”이 아니라 “행동 데이터”이다
- ScriptableObject + 데이터 중심 구조는 확장성이 매우 높다
- abstract / virtual은 역할이 명확할 때만 사용하는 것이 좋다
- BattleContext 같은 컨텍스트 객체는 시스템 결합도를 낮춰준다
- 중복 구조를 의심하고 제거하는 설계가 장기 유지보수에 중요하다
